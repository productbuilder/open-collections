## Current product context

TimeMap Collector is a Web Component named:

`timemap-collector`

It currently supports:
- asset cards in a collection grid
- card selection
- metadata editing in a sidebar
- source management via dialog
- manifest/export dialog
- multiple sources in progress or already implemented

The current UX still has these major issues:
- assets cannot be opened in a larger viewer
- the app can exceed the browser window height
- the collection viewport and metadata editor do not behave as proper independent scroll regions

This task should fix those issues.

## Phase 1 requirements

### 1. Responsive app layout

Refactor the main layout so the app fits within the browser window and behaves like a typical SaaS workspace.

Desired structure:

- fixed white header at the top
- main workspace below header
- workspace split into:
  - collection viewport on the left
  - metadata editor on the right

The app should fit inside the viewport rather than growing larger than the window.

### Required layout behavior

The overall app shell should behave approximately like:

- root component fills available height
- header remains visible
- main content area takes remaining height
- main content does not overflow the whole page vertically
- collection viewport scrolls independently
- metadata editor scrolls independently

A good implementation shape is:

- root app container:
  - `height: 100%` or `100vh` as appropriate
  - `display: flex`
  - `flex-direction: column`

- main workspace:
  - `flex: 1`
  - `display: flex`
  - `overflow: hidden`

- collection viewport:
  - `flex: 1` or equivalent wide area
  - `overflow-y: auto`

- metadata editor:
  - fixed or bounded width
  - `overflow-y: auto`

Make this work inside the Shadow DOM component.

Be careful not to break embedding behavior unnecessarily.

If `100vh` is too aggressive for embeddability, choose the cleanest approach that still makes the demo/app behave correctly in the browser and explain it briefly in the implementation notes.

---

### 2. Scrollable collection viewport

The left/main collection area should become a proper scroll container.

Requirements:
- the grid of cards must scroll vertically inside the collection area
- the entire page should not need to grow with the card list
- card layout should remain intact
- source badges / selection states / card interactions must continue to work

If helpful, add a small heading or toolbar area above the card grid inside the collection pane, but keep this minimal.

---

### 3. Scrollable metadata editor

The right metadata sidebar should also become a proper scroll container.

Requirements:
- metadata form scrolls independently from the card grid
- long metadata forms do not push the app height larger than the window
- current editable fields remain intact
- selected asset metadata still populates the sidebar

If practical, make the editor header area more stable, for example:
- title of selected item
- save/update affordance if one exists
- item/source badge if useful

Do not overdesign this.
Keep it simple and usable.

---

### 4. Asset viewer / detail view

Add a way to open an asset in a larger viewer.

This is a major part of Phase 1.

Requirements:
- user can open a selected asset into a larger detail view
- user can return to the card grid view
- viewer should work at least for image assets
- if other media types are not yet supported, fail gracefully

### Viewer interaction options
Implement one of these cleanly:

#### Preferred option
A modal/dialog/lightbox-style asset viewer

or

#### Acceptable option
An in-app detail panel/view state that replaces or overlays the grid temporarily

Choose whichever fits the current app structure best.

### Viewer should show
At minimum:
- large image preview
- title
- description if available
- source/provider info if available
- license if available

### Viewer controls
At minimum:
- close / back action
- escape key closes viewer if practical

Optional but nice:
- previous / next asset navigation
- open original media URL

Do not overbuild this if it complicates the current code too much.

### Card interaction
Make the interaction model clear.

A good pattern is:
- single click card = select asset
- double click card or dedicated open button = open viewer

Or:
- click thumbnail/open button = open viewer
- click card body = select asset

Choose the cleanest option based on the current implementation, but keep selection and opening understandable.
