Collection Registry (Registrator) and Collection Indexer

Collection Registry keeps track of what is known. Collection Indexer turns what is known into something searchable and explorable. WordPress can be one practical path for publishing and participating in that ecosystem.

Collection Registry in plain terms

Collection Registry is a lightweight persistent tool for tracking known publishers, collections, and registries. Registrator is the internal architectural module name.

It is not the same as Collection Indexer, and it does not need to start as a large, internet-scale backend.

  • Tracks what is known
  • Tracks validation and indexing state
  • Supports linked registries and federation

What Collection Registry stores

  • publishers
  • collection registrations
  • registry registrations
  • validation records
  • index jobs / indexing state references

Why Collection Registry exists

Collection owners, institutions, and communities need a practical way to register and keep track of known collections.

Registries can be local, thematic, institutional, national, or global. They can link to each other, which supports discovery without requiring one central platform.

Minimal persistence and deployment model

Recommended lightweight-first model:

  • SQLite-backed persistence
  • JSON export/import
  • optional published registry.json
  • no heavy database server required at first

Simplest deployment:

  • one small web app
  • one SQLite file on disk
  • one domain or subdomain
  • optional reverse proxy and scheduled backups

Implementation options

Go binary option

  • build Collection Registry as a Go application
  • distribute a standalone binary
  • prebuilt binaries avoid requiring users to install Go
  • SQLite fits naturally for file-based persistence

This is a strong self-hosted path with low infrastructure burden and easy backups.

Container option

  • run Collection Registry in a container
  • mount persistent storage for SQLite/data files
  • expose the UI/API through a domain

This avoids requiring Node or Go on the host, but does require a container runtime.

Hosted option (later)

A shared hosted registry service can exist later, but it is not required for the core design.

Collection Indexer relationship

Collection Registry keeps track of known things. Collection Indexer processes known collections.

Collection Indexer processes manifests, item metadata, media references, and creates derived browse/search structures.

They can run together in one early app deployment, while remaining separate in architecture and data model.

Linked registries and federation

  • a registry can register other registries
  • a registry can link to another registry
  • a registry can harvest summary data from another registry
  • a registry can optionally mirror another registry

When importing records from another registry, provenance should be preserved.

Indexing modes

  • registry-driven indexing
  • scheduled refresh of known records
  • optional broader discovery later

Starter data model and status vocabularies

See the full examples and starter status vocabularies in the technical reference: collection-registry-and-indexer.md.

Status sets:

  • Collection registration: draft, pending_validation, active, inactive, error
  • Validation: unknown, valid, warning, invalid
  • Indexing: not_indexed, pending, indexed, partial, failed
