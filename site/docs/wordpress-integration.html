WordPress integration

WordPress is one practical integration path for publishing, browsing, and lightweight registry participation in the Open Collections Protocol ecosystem.

WordPress is not the core architecture. It is one adoption path that can help people use the protocol through familiar CMS workflows.

Why WordPress matters

  • many institutions and communities already use WordPress
  • it lowers the barrier to adoption
  • it offers a familiar admin interface
  • it works well for smaller organizations and non-technical publishers

How WordPress fits the protocol

A plugin or integration should expose standard protocol outputs that other tools can consume, rather than making WordPress internals the public contract.

  • collection manifest (collection.json)
  • item URLs
  • media URLs
  • DCD discovery file (/.well-known/collections.json)
  • optional registry export (registry.json)

Possible WordPress roles

A) Publishing plugin

  • generate or expose collection.json
  • expose item detail resources
  • expose media references
  • optionally expose DCD discovery

B) Browser integration

  • embed Collection Browser views in WordPress pages
  • show published collections on institutional/public sites

C) Manager integration

  • embed Collection Manager into WordPress-admin workflows
  • support collection editing and publishing in familiar CMS contexts

D) Registry integration

  • allow WordPress to act as a lightweight Collection Registry
  • store known publishers, collections, and registries
  • publish a machine-readable registry export

Best first direction

A practical staged path is:

  • start with publishing collections
  • expose DCD
  • embed Collection Browser
  • optionally embed Collection Manager
  • add registry support later or as an extension

Relationship to Collection Registry and Collection Indexer

WordPress can be one implementation path for publishing and lightweight registry participation. Collection Registry remains a broader tool concept beyond WordPress, and Collection Indexer remains a separate processing layer.

Components and embedding

Collection Manager and Collection Browser can be used as embeddable web components, not only as standalone tools.

  • flexible attribute/configuration API
  • integration into pages, portals, and admin workflows
  • protocol outputs remain standard and reusable

Architectural guidance

  • keep protocol-facing outputs independent of WordPress internals
  • do not make WordPress-specific storage the public contract
  • use WordPress as an adapter/integration layer
  • preserve portable outputs and stable URLs
