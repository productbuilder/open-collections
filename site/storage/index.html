Storage options

Open Collections Protocol does not require one central platform. You can host collections with storage providers your team already knows.

The key requirement is a stable published web surface: predictable URLs for manifests, item details, and media.

Git providers

Git-based hosting works well when you want straightforward version history, collaborative editing workflows, and easy static publishing.

  • Good for: smaller-to-medium collections, team editing, audit trails.
  • Typical choices: GitHub, GitLab, Forgejo, and Gitea.
  • When to choose: when your team is comfortable with repository-based publishing.

S3-compatible object storage

S3-compatible storage is useful for larger media workloads and infrastructure setups that separate storage from application logic.

  • Good for: large asset volumes, scalable storage, CDN-friendly delivery.
  • Typical choices: AWS S3, Cloudflare R2, Backblaze B2 S3, and MinIO.
  • When to choose: when you need object-storage scale or already run S3-style infrastructure.

Choosing between Git and S3

Question Git providers S3-compatible providers
Do you want built-in version history and pull request workflows? Strong fit Usually handled by external tooling
Do you expect very large media storage needs? Possible, but may require extra planning Strong fit
Do you want simple static publishing from repository files? Strong fit Possible with static hosting layer in front

Publishing stability matters most

Whichever storage provider you choose, the public outcome should look the same: stable, crawlable, web-accessible collection resources.

That consistency allows Collection Manager, Collection Browser, indexers, registries, and other tools to interoperate without requiring one vendor platform.
